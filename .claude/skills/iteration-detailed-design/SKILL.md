# 需求迭代详设生成技能

走读版本计划文档和代码库，生成**需求详设文档**和**进度跟踪文档**。

---

## 执行前澄清机制

> ⚠️ 在开始任何步骤之前，必须先完成本节。禁止在信息不完整时单方面猜测生成。

### 第一阶段：执行前集中澄清

读取用户输入和计划文档后，识别以下两类不确定项，**集中在一条消息中一次性提问**：

**必须澄清（有歧义就问）：**

| 类型 | 需要确认的问题 | 示例场景 |
|------|---------------|----------|
| 需求描述歧义 | 功能边界可以理解为多种含义 | "支持导出"→ 导出 CSV 还是 PDF 还是两者都要？ |
| 技术偏好冲突 | 计划文档与用户口头说的技术方向不一致 | 文档写 REST，但用户之前提过想用 GraphQL |
| 版本范围不确定 | 不知道这次处理哪个版本或哪个模块 | "帮我做详设"但没说 V 几 |
| 计划文档内容矛盾 | 同一文档内有前后矛盾的描述 | 功能列表有 F-05，但开发步骤里没有对应 Step |

**不需要澄清（AI 自行合理假设并在文档中标注）：**

- 具体字段命名、接口参数细节
- 实现细节（用哪个库、内部算法选型）
- 文档格式和章节排序

**提问格式规范：**

```
在开始之前，有几点需要确认：

1. [问题，说明为什么需要确认]
2. [问题，说明为什么需要确认]
（如无歧义，说明"信息已足够，开始执行"，直接进入前置检查）
```

> 如果输入已足够清晰，跳过澄清直接执行，不要为了澄清而澄清。

### 第二阶段：执行后追问

文档产出完毕后，在独立的会话段落中列出生成过程中的剩余不确定项：

```
📌 文档已生成。以下内容生成时做了合理假设，请确认或修正：

1. [假设内容] → 我的处理方式是 [xxx]，如有不同请告知
2. [假设内容] → 我的处理方式是 [xxx]，如有不同请告知
```

> 若生成过程中无需假设，则不输出本段。

## 前置条件检查

> ⚠️ 澄清完成后，进入本节检查。任一硬性条件不满足，立即停止并告知用户缺什么。

### 输入起点判断

| 起点 | 条件 | 执行模式 |
|------|------|----------|
| **A：从零规划** | 有计划文档，代码库为空或极少 | 第二步扫描快速结束，第三步直接采用规划方案，跳过"已实现"对比列 |
| **B：迭代进行中** | 有计划文档，代码已部分实现 | 完整执行八步 |
| **C：逆向整理** | 只有代码，无计划文档 | ❌ 无法启动，见下方处理方式 |

### 硬性前置条件

**必须满足，缺一不可：**

1. **计划文档存在**：用户指定路径（默认 `docs/plan/`）下有目标版本的 markdown 文件
   - 文档必须包含：功能列表 + 至少一项数据模型或接口描述
   - 仅有标题或空文件视为不满足

2. **版本号明确**：用户已指定目标版本，不得自行猜测

**可选但影响执行深度：**

- 代码库可访问 → 决定第二步是否执行（起点 A 可跳过）
- 已有详设/进度文档 → 决定是**从头生成**还是**增量更新**（增量更新不覆盖已有内容）

### 起点 C 的处理方式

不能启动。提示用户：

- **选项 1**：先补写版本计划文档（可提供格式模板），再启动本 skill
- **选项 2**：改用代码逆向文档化流程（本 skill 不覆盖此场景）

### 计划文档格式检查

读取计划文档后，核查以下章节是否存在：

```
必须章节（缺失则提示补充，不强制中止）：
  □ 版本目标    → 至少一句说明本版本要达成什么
  □ 功能列表    → 有功能编号和优先级
  □ 数据模型设计 → 有表结构或字段说明
  □ 接口设计    → 有 Method + Path
  □ 开发步骤    → 有模块拆分
  □ 验收标准    → 每功能有可测试的条件
```

缺失章节的处理：在详设文档的对应位置标注 `⚠️ 计划文档未提供，需人工补充`，继续执行其余章节，不阻断整体流程。

### 检查通过后的执行声明

```
✅ 前置检查通过
起点类型：[A 从零规划 / B 迭代进行中]
目标版本：[version]
计划文档：[文件路径]
执行模式：[从头生成 / 增量更新]
```

## 触发条件

### 分级触发

**完整触发**（全部八步 + 产出文件）：

- "生成 V2 详设文档"
- "走读代码后给我出详设和进度文档"
- "对 V2.5 做完整详设"

**轻量触发**（仅分析输出，不产出文件）：

- "帮我看看这个版本计划合不合理"
- "这个功能怎么拆比较好"
- "V3 的需求拆分思路"

> 关键词"详设文档"、"需求拆分"、"进度文档"、"迭代规划"单独出现时，先确认触发级别再执行，不默认完整触发。

---

## 输入参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `version` | 目标版本号，如 V2、V2.5、V3 | 必填 |
| `plan_docs` | 计划文档路径 | `docs/plan/` |
| `scope` | 迭代范围：前置改动 / 核心模块 / 全部 | 全部 |

## 执行步骤

### 第一步：走读计划文档

读取计划文档，提取目标版本的功能列表、数据模型设计、接口设计、算法设计、开发步骤。

#### 实现状态判断规则

对每项功能，按以下优先级逐条核查，命中即停：

| 优先级 | 判断条件 | 标注状态 |
|--------|----------|----------|
| P1 | 有对应单测且通过 | ✅ 已实现 |
| P2 | API 路由注册 + Service 函数 + 前端调用链三者齐全 | ✅ 已实现 |
| P3 | 有函数体，但路由未注册**或**前端无调用 | 🔶 部分实现 |
| P4 | 只有注释 / TODO / 类型定义，无函数体 | 📋 未实现 |
| P5 | 计划文档有记录，代码库无任何痕迹 | 📋 未实现 |
| P6 | 函数体存在但逻辑明显不完整（如 `pass`、`TODO: impl`） | 🔧 需优化 |

> 同时命中多个级别时，取最低状态（P3 优先于 P2）。

---

### 第二步：分层扫描代码库

#### 扫描策略：三轮递进 + 硬性终止

**第一轮：目录骨架扫描**（不读文件内容）

- 读取根目录树状结构，识别框架约定（FastAPI / Next.js / Express 等）
- 定位入口文件：`main.py`、`router/index.ts`、`App.tsx`
- 产出：目录结构图 + 框架识别结论

**第二轮：关键词定位扫描**（最多读 20 个文件）

- 关键词来源：版本计划中的功能名、API 路径关键字、数据模型名
- 搜索范围：文件名匹配 → import 引用 → 函数名匹配
- 每个方向（后端 / 前端 / Agent）最多命中 7 个文件
- 超出限制：记录"未扫描文件清单"，在文档中标注"需人工补充"

**第三轮：函数签名级读取**（只读签名，不读函数体）

- 只提取：类名、函数名、参数列表、返回类型、装饰器
- 产出：文件路径清单 + 关键签名清单

#### 三个方向的扫描重点

| 方向 | 重点文件类型 |
|------|-------------|
| 后端 | 模型文件、Service 文件、API 路由、Ingest 入库、调度逻辑 |
| 前端 | 类型定义、API 函数、React Query Hooks、页面组件、路由配置 |
| Agent | 主入口、数据采集器、执行器、HTTP 端点、配置常量 |

> 扫描串行执行，按后端 → 前端 → Agent 顺序，不虚构"并行"。

---

### 第三步：对比分析，执行决策优先级

对每个功能点做规划方案 vs 已实现方案对比：

| 维度 | 规划方案 | 已实现方案 | 差异 | 选择及理由 |
|------|----------|------------|------|------------|

#### 决策优先级规则

| 优先级 | 场景 | 决策 |
|--------|------|------|
| P0 | 涉及平台层约定（技术选型、通信方向、全局 schema） | 无条件沿用已实现，不得替换 |
| P1 | 已实现且有线上验证记录 | 默认沿用，备注规划方案差异 |
| P2 | 规划方案有**可量化**的性能/安全收益 | 可替换，必须写明迁移步骤和回滚方案 |
| P3 | 纯代码风格差异 | 沿用已实现，不引入风格分歧 |
| P4 | 规划与已实现功能等价 | 沿用已实现，节省迁移成本 |

> 禁止用"规划方案更优雅"作为 P2 的理由，必须量化（如"减少 N 次 DB 查询"、"响应时间降低 X%"）。

---

### 第四步：建立架构边界

| 层级 | 定义 | 爆炸半径 | 策略 |
|------|------|----------|------|
| 实现层 | 模块内部逻辑 | 仅该模块 | 大胆尝试新理念 |
| 接口层 | 同模块内 Service 间调用 | 调用侧 Service | 可调整，需同步更新调用方 |
| 契约层 | API 路径 + schema + DB 表 | 跨模块消费者 | 保守确认，需兼容性分析 |
| 平台层 | 技术选型 + 通信方向 + 全局约定 | 全系统 | **不可变** |

依赖方向规则：只能从上层指向下层，严禁反向依赖。

---

### 第五步：拆分最小需求

将大功能拆为不可再分的最小需求，每拆出一个需求，必须通过原子性三问：

1. **独立上线问**：这个需求能独立上线，不依赖同批其他需求吗？否 → 继续拆，或将依赖显式标注为"前置需求"
2. **验收单句问**：验收标准能用一句话说清楚吗？否 → 继续拆
3. **并行开发问**：前后端能各自用 mock 独立开发吗？否 → 补齐接口契约后再拆

三问全部通过，才能进入详设编写。

每个最小需求必须包含：

1. **使用场景**：谁在什么时候做什么（一句话）
2. **后端设计**：数据模型 SQL + API 路径 + 请求/响应体 + 逻辑步骤 + 伪代码
3. **前端设计**：组件结构 + 类型定义 + API 函数 + Hook + 页面渲染
4. **Agent 设计**（如涉及）：端点 + 执行逻辑 + 上报数据结构
5. **联调校验**：函数签名对齐、请求体字段类型对齐、响应体解包路径对齐

---

### 第六步：产出详设文档

文件路径：`docs/plan/{version}-detailed-design.md`

**§0 架构边界与设计原则**：不可变层清单、可变层范围、依赖方向、兜底策略入口

**§0.5 变更影响分析**：
- 本次迭代预计修改的文件清单
- 每个文件的下游消费者（谁 import 了它）
- 回归测试建议：哪些已有功能需要重点验证
- DB 变更说明：是否有表结构改动，迁移脚本是否向后兼容（不兼容须标红）

**§N 各模块详设**：需求拆分表（含原子性验证结果）+ 每个最小需求的场景/后端/前端/Agent/联调五件套

**API 接口汇总与契约校验表**：全部新接口的 Method / Path / 请求体 / 响应体 / 关联需求编号

**前后端字段契约校验**：关键数据结构的字段名 / 类型 / 单位一致性验证

**方案对比与最佳实践选择**：每个有争议点的对比表 + 依据决策优先级的选择理由

**前端类型 / API / Hook 设计**：TypeScript interface 定义 + API 函数 + React Query Hooks

---

### 第七步：产出进度文档

文件路径：`docs/plan/{version}-progress.md`

必须包含：

- **状态图例**：📋 待开发 / 🔧 开发中 / ✅ 已完成 / 🚫 阻塞中
- **前置依赖清单**（标注阻塞关系和详设章节锚点）
- **各模块概览** + 子任务明细（编号 / 工作量 / 状态 / 备注）
- **依赖关系图**（ASCII 图，注明阻塞方向）
- **推荐开发顺序**（附理由）
- **工作量估算表**（后端 / 前端 / 联调分开统计，合计单人天 / 并行天）
- **兜底策略章节**

---

### 第八步：进度更新协议

#### 触发时机

- 用户说："xxx 完成了" / "xxx 合并了" / "xxx 上线了"
- 用户粘贴了 PR 链接或 commit message
- 用户说："把 N-xx 标记为完成"

#### 更新规则

1. 按**需求编号**匹配，不按文字匹配
2. 状态转移顺序：`📋 → 🔧 → ✅`，不允许跳级或逆转
3. 追加完成时间戳：`✅ 已完成（YYYY-MM-DD）`
4. 若实际实现与详设有偏差，备注列追加："实际与详设差异：[简述]"
5. 若需求被取消，标注 `❌ 已取消（原因：xxx）`，不删除该行

---

## 输出规范

- 语言与用户指令语言一致
- 伪代码：Python（后端）+ TypeScript（前端）
- SQL 用 MySQL 语法
- 所有接口响应体统一 `{ "data": ... }` 包装
- TypeScript 类型定义用 `interface`，字段名与后端 JSON key 完全一致
- 表格优先，避免长段落

---

## 兜底策略模板

对每个新模块 / 新设计，在详设文档中必须回答：

**1. 走不通的场景**：列出最可能失败的 2-3 个场景，必须具体到模块和失败原因，禁止写"可能有问题"。

**2. 兜底方式**：

| 场景 | 兜底方式 | 兜底后的最终状态 |
|------|----------|-----------------|
| 场景 A | 退回原方案 / 改协作接口 / 表迁移 / 删模块重来 | 描述最终可用状态 |

**3. 恢复周期**：每个场景的预估修复天数（单人）。

**4. 核心安全网**：明确陈述最坏情况下对已上线功能的影响为零。若不能保证，必须标注影响范围和隔离手段。

## 附：核心三项速查

**🔍 判断规则速查**

```
P1 有单测且通过              → ✅ 已实现
P2 路由+Service+前端调用链齐全 → ✅ 已实现
P3 有函数体但链路不完整       → 🔶 部分实现
P4 只有注释/TODO/类型定义     → 📋 未实现
P5 计划有，代码无             → 📋 未实现
P6 函数体不完整（pass/TODO）  → 🔧 需优化
```

**📐 扫描边界速查**

```
第一轮：目录结构 + 入口文件（无文件数限制）
第二轮：关键词定位，最多 20 个文件（每方向 ≤7）
第三轮：只读函数签名，不读函数体
超出限制：标注"未扫描"，人工补充
```

**⚖️ 决策优先级速查**

```
P0 平台层约定    → 无条件沿用已实现
P1 线上已验证    → 默认沿用
P2 可量化收益    → 可替换，需写迁移+回滚
P3 风格差异      → 沿用已实现
P4 功能等价      → 沿用已实现
禁止：以"更优雅"为由触发 P2
```
