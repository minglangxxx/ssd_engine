# Skill：需求详设编码生成

读取需求详设文档和项目代码库，按最小需求粒度逐个生成可运行代码，最终完成完整模块交付。

---

## 执行前澄清机制

> ⚠️ 在开始任何步骤之前，必须先完成本节。禁止在信息不完整时单方面猜测生成。

### 第一阶段：执行前集中澄清

读取用户输入和详设文档后，识别以下不确定项，**集中在一条消息中一次性提问**：

**必须澄清（有歧义就问）：**

| 类型 | 需要确认的问题 | 示例场景 |
|------|---------------|----------|
| 详设描述歧义 | 同一需求在详设中有多种可能实现路径 | "缓存用户状态"→ 存 Redis 还是存内存？ |
| 模块执行范围 | 不清楚这次要生成哪些模块或哪个版本 | "帮我编码"但没说 V 几或哪个模块 |
| 详设与代码冲突预判 | 扫描代码时发现已有实现与详设明显不一致 | 详设要求新表结构，但已有表字段不兼容 |

**不需要澄清（AI 自行合理假设并在输出后标注）：**

- 变量命名风格（沿用项目已有风格）
- 注释详略程度（沿用项目已有风格）
- 文件内部的代码组织顺序

**提问格式规范：**

```
在开始之前，有几点需要确认：

1. [问题，说明为什么需要确认]
2. [问题，说明为什么需要确认]
（如无歧义，说明"信息已足够，开始执行"）
```

> 如果输入已足够清晰，跳过澄清直接执行。

### 第二阶段：执行后追问

所有代码写入完毕后，在独立段落输出：

```
📌 编码已完成。以下内容生成时做了合理假设，请确认或修正：

1. [假设内容] → 我的处理方式是 [xxx]，如有不同请告知
```

> 若无假设，不输出本段。

---

## 前置条件检查

> ⚠️ 澄清完成后进入本节，任一硬性条件不满足立即停止并告知用户。

### 硬性前置条件

**必须满足，缺一不可：**

1. **详设文档存在**：`docs/plan/{version}-detailed-design.md` 可读取
   - 文档必须包含：§N 各模块详设 + API 接口汇总 + 前后端字段契约
   - 仅有标题或空章节视为不满足，提示用户先补充详设

2. **版本号明确**：用户已指定目标版本，不得自行猜测

3. **项目代码库可访问**：根目录可读取，用于识别技术栈和已有实现

**可选但影响执行深度：**

- 进度文档存在（`{version}-progress.md`）→ 用于跳过已完成需求，避免重复生成
- 测试文件存在 → 用于生成风格一致的单测

### 检查通过后的执行声明

```
✅ 前置检查通过
目标版本：[version]
详设文档：[文件路径]
项目根目录：[路径]
识别技术栈：[后端框架 / 前端框架 / 数据库]
执行范围：[模块列表]
```

## 触发条件

### 分级触发

**完整触发**（全流程 + 写入文件）：

- "根据详设生成 V2 的代码"
- "把 V2.5 详设编码实现"
- "生成 F-03 的代码"

**轻量触发**（仅分析或输出代码块，不写入文件）：

- "这个需求怎么实现比较好"
- "帮我看看这段代码思路对不对"
- "F-02 的后端逻辑怎么写"

> 完整触发前必须明确用户知晓"代码将直接写入项目文件"，若用户未明确说明，先确认再执行。

## 输入参数

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `version` | 目标版本号 | 必填 |
| `detailed_design` | 详设文档路径 | `docs/plan/{version}-detailed-design.md` |
| `project_root` | 项目根目录 | 当前工作目录 |
| `scope` | 执行范围：单需求 F-xx / 单模块 / 全部 | 全部 |
| `dry_run` | 仅输出代码块，不写入文件 | false |

## 执行步骤

### 第一步：识别技术栈

**不询问用户，直接走读项目代码识别。**

扫描以下文件确定技术栈：

| 扫描目标 | 识别内容 |
|----------|----------|
| `package.json` / `pyproject.toml` / `requirements.txt` | 依赖库和版本 |
| `main.py` / `app.py` / `index.ts` / `server.ts` | 框架类型（FastAPI / Express / NestJS 等） |
| `prisma/schema.prisma` / `models/` / `entities/` | ORM 类型和数据库 |
| `src/api/` / `src/hooks/` / `src/components/` | 前端架构模式 |
| 任意已有 Service 文件 | 函数签名风格、错误处理模式、响应体包装方式 |
| 任意已有单测文件 | 测试框架和测试风格 |

产出：**技术栈识别报告**（写入执行声明，不单独产出文件）

```
后端：[框架] + [ORM] + [数据库]
前端：[框架] + [状态管理] + [请求库]
测试：[后端测试框架] + [前端测试框架]
响应体格式：[示例，如 { "data": ... }]
错误处理模式：[示例]
命名风格：[snake_case / camelCase / 混用说明]
```

---

### 第二步：读取详设文档，建立需求执行队列

从详设文档提取所有最小需求，建立执行队列：

```
需求队列（示例）：
F-01 用户登录        [后端 + 前端]  状态：📋 待执行
F-02 任务创建        [后端 + 前端 + Agent]  状态：📋 待执行
F-03 任务列表查询    [后端 + 前端]  状态：📋 待执行
```

#### 需求过滤规则

若进度文档存在，按以下规则过滤队列：

| 进度文档状态 | 处理方式 |
|-------------|----------|
| ✅ 已完成 | 跳过，不重新生成 |
| 🔧 开发中 | 扫描已有代码，判断补全还是重写 |
| 📋 待开发 | 正常执行 |
| 🚫 阻塞中 | 跳过，在输出报告中标注原因 |

若进度文档不存在，扫描代码库，用**实现状态判断规则**（见下方）确定每个需求的初始状态。

#### 实现状态判断规则

| 优先级 | 判断条件 | 标注状态 |
|--------|----------|----------|
| P1 | 有对应单测且通过 | ✅ 跳过 |
| P2 | API 路由注册 + Service 函数 + 前端调用链三者齐全 | ✅ 跳过 |
| P3 | 有函数体，但路由未注册**或**前端无调用 | 🔶 补全 |
| P4 | 只有注释 / TODO / 类型定义，无函数体 | 📋 新建 |
| P5 | 详设有记录，代码库无任何痕迹 | 📋 新建 |
| P6 | 函数体存在但逻辑明显不完整 | 🔧 修改 |

---

### 第三步：逐需求生成代码（核心循环）

**对队列中每个需求，按以下子步骤执行，全部完成后才进入下一个需求。**

#### 子步骤 3-1：读取该需求的详设

从详设文档提取该需求的：

- 使用场景
- 后端设计（数据模型 SQL + API 路径 + 请求/响应体 + 逻辑步骤 + 伪代码）
- 前端设计（组件结构 + 类型定义 + API 函数 + Hook + 页面渲染）
- Agent 设计（如涉及）
- 联调校验要求

#### 子步骤 3-2：扫描该需求的相关现有文件

定位并读取与该需求直接相关的已有文件（最多 10 个），提取：

- 现有函数签名
- 现有数据模型字段
- 现有路由注册情况
- 与该需求有 import 关系的文件列表

#### 子步骤 3-3：冲突检测

对比详设要求与现有代码，检测以下三类冲突：

| 冲突类型 | 定义 | 处理方式 |
|----------|------|----------|
| **硬冲突** | 详设要求的字段/接口与已有实现不兼容，修改会破坏其他功能 | 暂停本需求及所有引用该路径的需求，记入冲突报告，继续执行其他需求 |
| **软冲突** | 详设与已有实现逻辑不同，但可以安全覆盖 | AI 自行判断，选择详设方案，在执行后追问中标注 |
| **无冲突** | 新增代码，或对已有代码的安全补全 | 正常执行 |

**硬冲突暂停格式：**

```
⛔ 需求 F-xx [功能名] 检测到硬冲突，已暂停：
  - 冲突位置：[文件路径:行号或函数名]
  - 冲突描述：[详设要求 xxx，但已有实现 yyy，修改会影响 zzz]
  - 受影响的需求：[F-xx, F-xx]（这些需求也暂停）
  继续执行其他需求...
```

#### 子步骤 3-4：生成代码

按以下顺序生成，保证依赖方向正确：

```
数据模型（Model）
    ↓
数据库迁移脚本（Migration）
    ↓
Service 层（业务逻辑）
    ↓
API 路由层（Controller/Router）
    ↓
前端类型定义（TypeScript interface）
    ↓
前端 API 函数
    ↓
前端 Hook（React Query 或项目使用的状态管理）
    ↓
前端组件/页面
    ↓
单元测试（后端 Service 层）
```

**代码生成约束：**

- 命名风格、注释风格、错误处理模式完全沿用第一步识别的项目风格
- 响应体格式与项目已有接口完全一致，不自行发明新格式
- 禁止引入详设文档未提及的新依赖库
- 禁止修改与本需求无关的已有代码
- 每个生成的函数必须有完整实现，禁止出现 `pass`、`TODO`、`// implement me`

#### 子步骤 3-5：写入文件

按以下规则写入：

| 情况 | 操作 |
|------|------|
| 文件不存在 | 创建新文件，写入完整内容 |
| 文件存在，新增函数/类 | 在文件末尾或约定位置追加，不改动已有内容 |
| 文件存在，修改已有函数 | 只替换目标函数体，保留文件其他内容 |
| 文件存在，硬冲突 | 不写入，记入冲突报告 |

写入后输出确认：

```
✅ F-01 用户登录 代码已写入：
  后端：
    - app/models/user.py （新建）
    - app/services/auth_service.py （新增函数 login）
    - app/routers/auth.py （新增路由 POST /api/auth/login）
    - tests/test_auth_service.py （新建）
  前端：
    - src/types/auth.ts （新增 interface LoginRequest, LoginResponse）
    - src/api/auth.ts （新增函数 login）
    - src/hooks/useLogin.ts （新建）
    - src/pages/LoginPage.tsx （新建）
```

#### 子步骤 3-6：更新进度文档

若进度文档存在，写入完成后立即更新对应行：

```
F-01 → ✅ 已完成（YYYY-MM-DD）
```

若本需求暂停，更新为：

```
F-xx → 🚫 阻塞中（硬冲突，见冲突报告）
```

---

### 第四步：模块内联调验证

**所有需求代码生成完毕后，在代码层面做联调校验，不运行代码，只做静态分析。**

对每对前后端接口，验证：

| 校验项 | 验证方式 |
|--------|----------|
| 接口路径一致 | 前端 API 函数调用的路径 = 后端路由注册的路径 |
| 请求体字段一致 | 前端 interface 字段名/类型 = 后端接受的参数名/类型 |
| 响应体解包路径一致 | 前端解包方式（如 `res.data.xxx`）= 后端响应体结构 |
| 必填字段无遗漏 | 后端标注必填的字段，前端请求体中均有对应 |

发现不一致时，直接修正代码后重新写入，不暂停流程，在执行后追问中标注修正内容。

---

### 第五步：输出完整执行报告

所有需求处理完毕后，输出一份执行报告：

```markdown
## 编码执行报告 - {version}

### 完成情况概览

| 需求 | 功能名 | 状态 | 写入文件数 |
|------|--------|------|------------|
| F-01 | 用户登录 | ✅ 已完成 | 8 |
| F-02 | 任务创建 | ✅ 已完成 | 6 |
| F-03 | 任务列表 | ⛔ 硬冲突暂停 | 0 |

### 冲突报告（需用户决策）

#### F-03 任务列表查询

**冲突位置**：`app/models/task.py` - `Task` 模型

**详设要求**：
  新增字段 `priority: int`，默认值 0

**已有实现**：
  `priority` 字段已存在，类型为 `str`（枚举值 'low'/'medium'/'high'），
  被 `app/services/task_service.py` 和 `app/routers/task.py` 引用

**修改影响**：
  修改字段类型会破坏 task_service.py 中的 3 处枚举比较逻辑，
  以及前端 `src/types/task.ts` 中的 `TaskPriority` 类型定义

**受影响的需求**：F-03、F-05（任务筛选，依赖 priority 字段）

**请选择处理方式**：
  A. 修改详设，沿用已有 str 类型的 priority 字段
  B. 修改已有实现，将 priority 改为 int（需同步修改 task_service、前端类型）
  C. 跳过 priority 字段，其余部分正常生成

### 假设说明（需用户确认）

1. [假设内容] → 处理方式 [xxx]
```

## 代码质量约束

### 必须遵守

- 每个函数有完整实现，无占位符
- 命名、注释、错误处理完全沿用项目已有风格
- 不引入详设未提及的新依赖
- 不修改与本需求无关的已有代码
- 数据库迁移脚本必须向后兼容（新增字段用 DEFAULT 或 NULLABLE）
- 单测覆盖 Service 层的核心逻辑，至少包含正常路径和主要异常路径

### 禁止行为

- 生成 `TODO`、`pass`、`// implement me` 等占位符
- 自行发明新的响应体格式
- 跨需求边界修改代码（本需求的代码只影响本需求涉及的文件）
- 在未检测冲突的情况下直接覆盖已有函数
- 假设用户会手动处理某些步骤而省略生成

## 附：执行流程速查

```
[澄清] → [前置检查] → [识别技术栈]
    → [建立需求队列]
    → 对每个需求循环：
        读取详设
        → 扫描相关文件
        → 冲突检测
            → 硬冲突：暂停 + 记录，继续下一个
            → 软冲突/无冲突：生成代码
        → 写入文件
        → 更新进度文档
    → [联调校验]
    → [输出执行报告]
    → [执行后追问]
```

```
冲突处理原则：
硬冲突 → 暂停该需求及关联需求，记入报告，用户决策
软冲突 → AI 自行选择详设方案，执行后标注
无冲突 → 正常写入
```

```
写入原则：
新文件    → 创建
新增内容  → 追加到约定位置
修改内容  → 只替换目标函数，不动其他代码
冲突内容  → 不写入
```
