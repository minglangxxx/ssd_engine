# SSD Engine 数据采集与持久化重构方案

## 1. 文档目标

本文档用于指导 SSD Engine 后续围绕 FIO 数据、磁盘监控数据和 AI 分析数据的重构编码工作，重点回答以下问题：

1. 哪些数据需要持久化，谁负责写入。
2. 数据在活跃、归档、压缩三个阶段如何流转。
3. 磁盘监控如何从当前的 Agent 内存历史模式重构为 Backend 持久化查询模式。
4. FIO 结果、磁盘监控、AI 分析如何统一纳入数据管理功能。
5. 如何保证 AI 分析能在 7 天内稳定拉取 FIO 和磁盘监控数据。

本文档以当前仓库结构为前提，尽量复用现有 Backend、Frontend、Agent 分层，不引入新的独立基础设施。

## 2. 当前问题

### 2.1 Agent 端监控历史不可作为正式数据源

当前 Agent 的主机监控和磁盘监控历史数据保存在内存环形缓冲区中，只适合短时缓存，不满足正式数据留存需求。存在以下问题：

- Agent 重启后历史数据全部丢失。
- 无法支撑任意时间范围过滤和长期回查。
- 当前磁盘采样与查询逻辑耦合，容易影响样本口径。
- 数据管理功能无法统一接入 Agent 内存中的历史数据。

### 2.2 FIO 与监控数据没有统一生命周期模型

当前系统中：

- FIO 任务摘要保存在 `tasks.result`。
- FIO 趋势保存在 `fio_trend_data`。
- 磁盘监控历史并未正式落库。
- 数据管理依赖 `data_records`，但尚未覆盖全部数据资产。

结果是：

- 数据类型之间生命周期策略不一致。
- 数据管理页面无法统一展示热数据、归档数据、压缩数据。
- AI 分析输入范围无法稳定溯源。

### 2.3 AI 分析的数据依赖缺少正式边界

AI 分析需要读取任务窗口内的 FIO 数据和磁盘监控数据，但当前磁盘监控读取路径依赖 Agent 在线和内存历史。对于超过短期窗口的数据，没有稳定的数据读取保障。

## 3. 需求约束

本次重构需要满足以下目标：

1. FIO 任务测试结果需要持久化存储。
2. 超过指定活跃天数的 FIO 任务数据需要归档，超过指定归档天数的数据需要压缩。
3. 归档和压缩后的文件需要展示到数据管理功能。
4. 磁盘监控数据需要支持与 FIO 相同的数据生命周期处理。
5. AI 分析依赖 FIO 数据和磁盘监控数据，当前仅支持分析 7 天内的任务数据。

## 4. 总体设计原则

### 4.1 Backend 作为唯一正式查询源

重构后，Frontend 的 FIO 趋势、磁盘监控、数据管理、AI 分析统一从 Backend 读取。Agent 不再承担正式历史查询数据源职责。

### 4.2 Agent 只负责生产数据，不负责长期保存真相

Agent 的职责调整为：

- 执行 FIO 任务。
- 周期采集监控样本。
- 以批量方式推送样本到 Backend。
- 在 Backend 不可用时临时落本地缓冲，等待补传。

### 4.3 数据资产和原始样本分离管理

原始样本用于监控查询和 AI 分析，数据资产用于生命周期管理和文件展示。

- 原始样本存于热表。
- 归档文件和压缩文件通过 `data_records` 管理。

### 4.4 生命周期规则统一但粒度不同

不同数据类型使用统一状态机：

- `active`
- `archived`
- `compressed`

但归档粒度不同：

- FIO 结果和 FIO 趋势按任务归档。
- 磁盘监控按节点、磁盘、日期归档。
- AI 分析按任务归档。

```text
Frontend
   |

   v

   |

   v

1. 热数据表
```

- Agent 负责写入样本到 Backend。
- Backend 负责校验、落库、归档、压缩、查询。
- Frontend 不直接读取 Agent 历史数据。

## 7. 表结构设计

## 7.1 tasks 表增强

现有 `tasks` 表继续保留，作为 FIO 任务主表。

建议新增字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `started_at` | DATETIME | FIO 实际开始时间 |
| `finished_at` | DATETIME | FIO 实际结束时间 |
| `data_window_start` | DATETIME | 默认数据分析窗口开始时间 |
| `data_window_end` | DATETIME | 默认数据分析窗口结束时间 |
| `retention_policy` | JSON | 任务级生命周期策略快照 |
| `last_analysis_at` | DATETIME | 最近一次 AI 分析时间 |

说明：

- `tasks.result` 继续保存任务摘要结果，避免每次详情页都聚合趋势点。
- `started_at`、`finished_at` 作为 FIO 和磁盘监控的统一时间锚点。

## 7.2 fio_trend_data 表增强

现有 `fio_trend_data` 表继续保留，用于存储热数据期内的 FIO 趋势点。

建议新增字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `device_ip` | VARCHAR(50) | 设备节点 IP |
| `device_path` | VARCHAR(255) | 测试目标设备路径 |

| `source` | VARCHAR(32) | 数据来源，固定为 `agent_fio` |
| `created_at` | DATETIME | 入库时间 |

建议索引：

- `(task_id, timestamp)`

## 7.3 新增 disk_monitor_samples 表

该表是磁盘监控正式查询的热数据表。

建议建表：

```sql
CREATE TABLE disk_monitor_samples (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    device_ip VARCHAR(50) NOT NULL,
    disk_name VARCHAR(64) NOT NULL,
    event_time DATETIME(3) NOT NULL,
    task_id INT NULL,
    sample_interval_ms INT NOT NULL DEFAULT 1000,
    disk_iops_read DOUBLE NOT NULL DEFAULT 0,
    disk_iops_write DOUBLE NOT NULL DEFAULT 0,
    disk_bw_read_bytes_per_sec DOUBLE NOT NULL DEFAULT 0,
    disk_bw_write_bytes_per_sec DOUBLE NOT NULL DEFAULT 0,
    disk_latency_read_ms DOUBLE NOT NULL DEFAULT 0,
    disk_latency_write_ms DOUBLE NOT NULL DEFAULT 0,
    disk_queue_depth DOUBLE NOT NULL DEFAULT 0,
    disk_await_ms DOUBLE NOT NULL DEFAULT 0,
    disk_svctm_ms DOUBLE NOT NULL DEFAULT 0,
    disk_util_percent DOUBLE NOT NULL DEFAULT 0,
    disk_rrqm_per_sec DOUBLE NOT NULL DEFAULT 0,
    disk_wrqm_per_sec DOUBLE NOT NULL DEFAULT 0,
    source VARCHAR(32) NOT NULL DEFAULT 'agent_disk',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_disk_time (device_ip, disk_name, event_time),
    INDEX idx_task_time (task_id, event_time),
    INDEX idx_event_time (event_time)
);
```

说明：

- `task_id` 可空。
- 正常背景监控点 `task_id` 为空。
- 若后续需要将任务执行窗口内的监控点与任务建立显式关联，可在写入或离线任务中回填 `task_id`。

## 7.4 ai_analyses 表增强

## 8. 生命周期状态机

所有数据资产统一使用以下状态流转：

```text
active -> archived -> compressed
```

### 8.1 active

定义：

- 数据仍存储在热表中，可直接查询。
- 对应 DataRecord 可选登记，也可在写入完成时立即登记。

### 8.2 archived

定义：

- 数据已经从热表导出为归档文件。
- DataRecord 的 `file_path` 指向归档文件。
- 数据管理页面可直接下载。

### 8.3 compressed

定义：

- 归档文件已被压缩。
- DataRecord 的 `compressed_path` 指向压缩包。
- 未压缩文件可按策略保留或删除。

## 9. 数据文件组织规范

建议在 Backend 的 `data/` 目录下采用以下组织方式：

```text
backend/data/
├── active/
├── archive/
│   ├── fio/
│   │   └── task_<task_id>/
│   ├── disk/
│   │   └── <device_ip>/<disk_name>/YYYYMMDD/
│   └── analysis/
│       └── task_<task_id>/
└── compressed/
    ├── fio/
    ├── disk/
    └── analysis/
```

建议归档文件格式：

- FIO 结果摘要：JSON
- FIO 趋势：JSONL 或 Parquet
- 磁盘监控：JSONL 或 Parquet
- AI 分析：Markdown + JSON 摘要

## 10. Agent 采集链路设计

## 10.1 Agent 职责调整

Agent 重构后负责：

1. 启停 FIO 任务。
2. 固定周期采集磁盘监控样本。
3. 批量推送 FIO 趋势点到 Backend。
4. 批量推送磁盘监控样本到 Backend。
5. Backend 不可用时落本地 spool，等待补传。

Agent 不再负责：

1. 正式提供历史监控查询。
2. 作为前端磁盘监控页面的数据真相来源。

## 10.2 FIO 趋势采集流程

```text
FIO 运行 -> FioRunner 产出趋势点 -> Agent 批量缓存 -> POST Backend ingest API -> Backend 落 fio_trend_data
```

建议规则：

- 默认 1 秒采一个趋势点。
- 每 3 到 5 秒批量推送一次。
- 任务结束时补齐最后一批。
- 每个批次携带 `task_id`、`device_ip`、`device_path`。

## 10.3 磁盘监控采集流程

```text
MonitorSampler 每秒采样 -> 生成 disk_monitor_samples 点 -> Agent 批量缓存 -> POST Backend ingest API -> Backend 落 disk_monitor_samples
```

建议规则：

- 使用独立监控线程维护采样状态。
- 不允许查询接口推进差分基线。
- 差分基线仅由后台采样线程持有。
- 若 Backend 不可用，将批次落地到本地 spool 文件。

## 10.4 Agent 本地缓冲设计

建议新增 `agent/spool/` 目录，用于短时故障补传。

建议文件组织：

```text
agent/spool/
├── fio_trend/
└── disk_monitor/
```

建议行为：

- 写入失败时，批次序列化为 JSON 文件。
- 后台补传线程周期扫描并重试。
- 补传成功后删除本地文件。

## 11. Backend 写入链路设计

## 11.1 新增内部写入接口

建议新增内部 API，仅供 Agent 调用：

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/internal/ingest/fio-trend` | 批量写入 FIO 趋势点 |
| POST | `/api/internal/ingest/disk-monitor` | 批量写入磁盘监控点 |
| POST | `/api/internal/ingest/flush-task` | 任务结束后补齐元数据 |

建议鉴权方式：

- Agent Token
- 或基于设备维度预共享密钥

## 11.2 FIO 趋势写入服务

建议新增服务：

- `backend/app/services/ingest_service.py`

职责：

1. 校验任务是否存在。
2. 校验设备 IP 与任务设备是否一致。
3. 批量写入 `fio_trend_data`。
4. 更新任务 `started_at`、`finished_at`。
5. 同步或更新 `data_records` 中的 `fio_trend` 资产。

## 11.3 磁盘监控写入服务

职责：

1. 校验设备是否已登记。
2. 批量写入 `disk_monitor_samples`。
3. 为对应的 `device_ip + disk_name + date` 维护 `data_records` 资产。
4. 记录 `window_start`、`window_end`、`record_count`、`size_bytes`。

## 12. 查询链路设计

## 12.1 FIO 趋势查询

现有外部接口可保持不变：

- `GET /api/tasks/<id>/trend`

但实现改为：

1. 只查询 Backend `fio_trend_data`。
2. 运行中任务也以 Backend 热表为准。
3. 不再优先从 Agent 内存 deque 获取。

## 12.2 磁盘监控查询

现有外部接口可保持不变：

- `GET /api/monitor/hosts/<host>/disks/<disk>/metrics`

但实现改为：

1. 若时间范围落在热数据窗口内，直接查询 `disk_monitor_samples`。
2. 若时间范围跨越归档窗口，查询热表并读取归档文件后合并。
3. 若范围仅落在压缩层，读取压缩包内容并返回。

查询优先级：

```text
热表 -> 归档文件 -> 压缩文件
```

## 12.3 数据管理查询

`GET /api/data` 需要扩展支持以下筛选：

- `task_id`
- `disk_name`
- `window_start`
- `window_end`
- `data_type`
- `status`

数据管理页面展示建议补充以下列：

- 覆盖时间范围
- 记录条数
- 存储格式
- 文件状态
- 下载入口

## 12.4 AI 分析查询

AI 分析仅读取热数据，不读取归档和压缩文件。

规则：

1. 仅允许分析 `finished_at` 在 7 天内的任务。
2. FIO 数据从 `tasks` 和 `fio_trend_data` 获取。
3. 磁盘监控数据从 `disk_monitor_samples` 获取。
4. 超过 7 天直接返回不支持分析。

## 13. AI 分析约束设计

## 13.1 分析时间范围

分析窗口来源：

1. 优先使用 `tasks.started_at` 和 `tasks.finished_at`。
2. 若缺失，则回退到 `fio_trend_data` 的首尾点。
3. 在此基础上加前后扩展窗口。

## 13.2 分析输入清单

每次分析需记录：

- 任务 ID
- 任务设备路径
- FIO 趋势时间范围
- 磁盘监控时间范围
- 使用的数据点数量
- 数据来源版本

记录到 `ai_analyses.input_manifest`。

## 14. 生命周期任务设计

建议继续使用 APScheduler，在 `backend/app/services/data_lifecycle.py` 中统一编排。

## 14.1 任务一：热数据归档

执行频率：每日一次。

处理规则：

### FIO 数据

- 找出 `finished_at` 早于活跃保留阈值的任务。
- 导出任务摘要和趋势点。
- 写归档文件。
- 创建或更新 `data_records`。
- 删除过期热表趋势点。

### 磁盘监控数据

- 按 `device_ip + disk_name + 日期` 找出早于活跃保留阈值的数据块。
- 导出为归档文件。
- 创建或更新 `data_records`。
- 删除对应热表样本。

## 14.2 任务二：归档压缩

执行频率：每日一次。

处理规则：

- 找出 `archived_at` 早于归档保留阈值的 `data_records`。
- 将文件压缩为 tar.gz 或 zstd 包。
- 更新 `compressed_path`、`compressed_at`、`status`。

## 14.3 任务三：元数据校准

执行频率：每日一次。

处理规则：

- 校准 `record_count`、`size_bytes`、`window_start`、`window_end`。
- 清理缺失文件或异常记录。

## 15. 配置项建议

建议在 Backend 配置中新增以下参数：

| 配置项 | 说明 | 默认建议 |
|------|------|------|
| `FIO_ACTIVE_DAYS` | FIO 热数据保留天数 | 7 |
| `FIO_ARCHIVE_DAYS` | FIO 归档后进入压缩的天数 | 30 |
| `DISK_MONITOR_ACTIVE_DAYS` | 磁盘监控热数据保留天数 | 7 |
| `DISK_MONITOR_ARCHIVE_DAYS` | 磁盘监控归档后进入压缩的天数 | 30 |
| `AI_ANALYSIS_MAX_DAYS` | AI 分析允许回看的任务最大天数 | 7 |
| `INGEST_BATCH_SIZE` | Agent 批量推送最大点数 | 100 |
| `INGEST_RETRY_INTERVAL` | Agent 补传重试间隔 | 10s |

## 18. 本轮编码优先级建议

若后续按最短交付路径推进，建议编码顺序如下：

1. 新增 `disk_monitor_samples` 表和模型。
2. 新增 Agent 到 Backend 的 ingest API。
3. 重写磁盘监控查询链路，停用正式页面对 Agent 历史接口的依赖。
4. 重写 FIO 趋势查询链路，统一改为 Backend 热表。
5. 扩展 `data_records` 并接入数据管理页面。
6. 实现归档任务和压缩任务。
7. 最后收口 AI 分析的 7 天限制与输入清单。

## 19. 结论

本次重构的核心不是单独给磁盘监控补一个落库表，而是建立一套统一的数据采集、持久化、生命周期和查询闭环：

- Agent 只负责生产数据和补传。
- Backend 成为唯一正式查询源。
- FIO、磁盘监控、AI 分析统一纳入数据资产管理。
- 数据管理统一展示热数据、归档数据、压缩数据。
- AI 分析只基于 7 天热数据，避免跨冷热层读取带来的复杂性。

该方案兼容当前项目结构，能够在不引入外部时序数据库的前提下，先完成当前阶段的数据治理目标，并为后续扩展保留空间。

## 26. 风险与注意事项

1. 第一版不要同时引入外部 TSDB，否则会扩大改造面。
2. 第一版归档格式优先选 JSONL，便于调试和人工排查。
3. `disk_monitor_samples` 数据量会较大，必须尽早按时间做归档清理。
4. Agent 到 Backend 的 ingest 必须具备幂等容忍，避免重试时产生异常重复点。
5. AI 分析严格限制只读热数据，避免归档读取链路阻塞交付。

## 27. 后续编码建议顺序

为降低重构风险，建议按以下顺序编码：

1. 新建 `disk_monitor_samples` 模型和 migration。
2. 新建 internal ingest API 和 ingest service。
3. 改造 Agent 的 FIO 趋势和磁盘监控上报链路。
4. 重写 `monitor_service` 和 `task_service.get_trend`。
5. 扩展 `data_records` 与数据管理页面。
6. 实现归档、压缩和下载。
7. 最后收口 AI 分析 7 天限制与输入清单。
