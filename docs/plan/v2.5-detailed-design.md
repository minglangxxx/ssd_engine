## §0 架构边界与设计原则

### §0.1 不可变层清单

| 层级 | 约定 | 说明 |
|------|------|------|
| 平台层 | Flask + SQLAlchemy + MySQL + APScheduler + threading | 与 V1/V2 一致，不引入 Celery/Redis |
| 平台层 | Agent-Push 心跳 + Server 主动推送任务 | 与 V1/V2 调度模式一致 |
| 平台层 | 前端 React + TypeScript + Ant Design + React Query + ECharts | 技术栈不变 |
| 平台层 | API 统一 `{ "data": ... }` 响应包装 | 不变 |
| 平台层 | NVMe 磁盘名归一化 `_normalize_nvme_disk_name()` / `_extract_nvme_controller()` | 与 V1/V2 入库/查询一致 |

### §0.2 可变层范围

| 层级 | 范围 | 策略 |
|------|------|------|
| 契约层 | `nvme_test` 表结构、API 路径、请求/响应 Schema | 本次新增，需兼容性设计 |
| 接口层 | NvmeService 内部拆分出 ValidationService | 可调整，同步更新调用方 |
| 实现层 | 校验规则引擎、结果解析、前端组件 | 大胆尝试 |

### §0.3 依赖方向

```
前端 → API 契约层 → Service 层 → Model 层 → DB
                  ↘ AgentExecutor → Agent
```

严禁反向依赖。Service 不可直接依赖前端类型定义，Model 不可调用 Service。

### §0.4 兜底策略入口

本节纲要，各模块详设中有独立的兜底章节。

| 场景 | 兜底方式 | 最终状态 |
|------|----------|---------|
| 校验规则引擎复杂度过高 | 退回 if-else 硬编码规则，逐字段校验 | 功能可用，规则不可配置 |
| Error Log 三步流程超时 | 跳过触发步骤，直接 PASS 并标注"未验证" | 报告标注，不影响其他模块 |
| Agent 离线导致命令失败 | 单项 FAIL + error 字段记录，其他项继续 | 部分结果可用 |

## §1 通用校验规则引擎

### §1.1 需求拆分

| 编号 | 最小需求 | 原子性验证 | 前置需求 |
|------|---------|-----------|---------|
| VE-1 | 定义校验规则数据结构 | 独立上线：规则可被加载和执行 | 无 |
| VE-2 | 字段校验执行器：对单个字段应用规则返回 Pass/Fail | 验收：给定值和规则，返回正确布尔+reason | VE-1 |
| VE-3 | 结构化结果输出：所有字段结果聚合为 verdict | 验收：全 Pass→PASS，有 Fail→FAIL，有 Warn→PARTIAL | VE-2 |
| VE-4 | 规则可配置化（YAML/JSON 文件） | 验收：修改规则文件不需要改代码 | VE-1 |

```python
# backend/app/services/nvme_validation_service.py

from dataclasses import dataclass
from enum import Enum
from typing import Any

class CheckOp(Enum):
    NOT_EMPTY = "not_empty"
    NOT_ZERO = "not_zero"
    GTE = "gte"        # >= threshold
    LTE = "lte"        # <= threshold
    RANGE = "range"    # [lo, hi]
    REGEX = "regex"    # 正则匹配
    IN_SET = "in_set"  # 值在集合内
    LEN_LTE = "len_lte"  # 字符串长度 <=
    LEN_GTE = "len_gte"  # 字符串长度 >=
    INCREASED = "increased"  # after > before（Error Log 专用）
    FORMAT = "format"  # 0x+4位十六进制（VID专用）

@dataclass
class CheckRule:
    field: str            # JSON key 路径，如 "vid", "lbafs[0].ds"
    check: CheckOp
    label: str            # 中文描述，如 "厂商ID"
    params: dict | None = None  # {"threshold": 1, "regex": r"^0x[0-9a-fA-F]{4}$"}
    level: str = "fail"   # fail / warn

@dataclass
class CheckResult:
    field: str
    value: Any
    check: str            # 规则描述
    pass_: bool           # Python 关键字回避
    reason: str           # 失败原因或"通过"
    level: str            # fail / warn

def evaluate_rule(value: Any, rule: CheckRule) -> CheckResult:
    """对单个字段执行校验"""
    passed = True
    reason = "通过"

    if rule.check == CheckOp.NOT_EMPTY:
        if value is None or value == "":
            passed = False
            reason = f"{rule.label}为空"
    elif rule.check == CheckOp.NOT_ZERO:
        if not value or int(value) == 0:
            passed = False
            reason = f"{rule.label}为0"
    elif rule.check == CheckOp.GTE:
        threshold = rule.params.get("threshold", 1)
        if value is None or int(value) < threshold:
            passed = False
            reason = f"{rule.label}={value}，要求≥{threshold}"
    elif rule.check == CheckOp.LTE:
        threshold = rule.params.get("threshold", 70)
        if value is not None and int(value) > threshold:
            passed = False
            reason = f"{rule.label}={value}，要求≤{threshold}"
    elif rule.check == CheckOp.RANGE:
        lo, hi = rule.params.get("lo", 0), rule.params.get("hi", 100)
        if value is None or not (lo <= int(value) <= hi):
            passed = False
            reason = f"{rule.label}={value}，要求在[{lo},{hi}]范围"
    elif rule.check == CheckOp.REGEX:
        import re
        pattern = rule.params.get("regex", "")
        if not re.match(pattern, str(value or "")):
            passed = False
            reason = f"{rule.label}格式不匹配"
    elif rule.check == CheckOp.IN_SET:
        allowed = rule.params.get("set", [])
        if str(value) not in [str(v) for v in allowed]:
            passed = False
            reason = f"{rule.label}={value}，不在允许集合内"
    elif rule.check == CheckOp.LEN_LTE:
        max_len = rule.params.get("max", 20)
        if value and len(str(value)) > max_len:
            passed = False
            reason = f"{rule.label}长度{len(str(value))}，超过{max_len}"
    elif rule.check == CheckOp.LEN_GTE:
        min_len = rule.params.get("min", 1)
        if not value or len(str(value)) < min_len:
            passed = False
            reason = f"{rule.label}长度不足{min_len}"
    elif rule.check == CheckOp.FORMAT:
        if rule.params.get("fmt") == "hex4":
            import re
            if not re.match(r"^0x[0-9a-fA-F]{4}$", str(value or "")):
                passed = False
                reason = f"{rule.label}格式需为0x+4位十六进制"
    elif rule.check == CheckOp.INCREASED:
        # Error Log 专用，params 中需有 before/after
        before = rule.params.get("before", 0)
        after = rule.params.get("after", 0)
        if after <= before:
            passed = False
            reason = f"错误日志条数未增加（{before}→{after}）"
        else:
            reason = f"错误日志正确增加（{before}→{after}）"

    return CheckResult(
        field=rule.field,
        value=value,
        check=rule.label,
        pass_=passed,
        reason=reason,
        level=rule.level,
    )


def aggregate_verdict(results: list[CheckResult]) -> str:
    """聚合判定：fail→FAIL, warn→PARTIAL, all pass→PASS"""
    has_fail = any(not r.pass_ and r.level == "fail" for r in results)
    has_warn = any(not r.pass_ and r.level == "warn" for r in results)
    if has_fail:
        return "FAIL"
    if has_warn:
        return "PARTIAL"
    return "PASS"
```

```python
# NvmeValidationService 核心方法签名

class NvmeValidationService:
    @staticmethod
    def run_validation(device_id: int, disk_name: str, test_type: str) -> dict:
        """创建 nvme_test 记录 → 从 Agent 获取数据 → 执行规则 → 写入结果 → 返回"""

    @staticmethod
    def get_validation_result(test_id: int) -> dict:
        """按 ID 查询校验结果"""

    @staticmethod
    def list_validations(device_id: int, disk_name: str | None, test_type: str | None) -> dict:
        """列表查询，支持筛选"""

    @staticmethod
    def load_rules(test_type: str) -> list[CheckRule]:
        """从 YAML 文件加载校验规则"""

    @staticmethod
    def apply_rules(raw_data: dict, rules: list[CheckRule]) -> list[CheckResult]:
        """对原始数据执行全部规则"""
```

#### 伪代码：run_validation

```python
def run_validation(device_id, disk_name, test_type):
    device = Device.query.get(device_id)
    if not device:
        raise ApiError('NOT_FOUND', '设备不存在', 404)

    # 1. 创建 nvme_test 记录
    test = NvmeTest(device_id=device.id, disk_name=disk_name,
                    test_type=test_type, status='pending')
    db.session.add(test)
    db.session.commit()

    # 2. 启动 daemon thread 执行校验
    thread = threading.Thread(
        target=NvmeValidationService._run_in_background,
        args=(test.id, device, disk_name, test_type),
        daemon=True
    )
    thread.start()

    return {'test_id': test.id, 'status': 'pending'}


def _run_in_background(test_id, device, disk_name, test_type):
    app = current_app._get_current_object()
    with app.app_context():
        try:
            test = NvmeTest.query.get(test_id)
            test.status = 'running'
            db.session.commit()

            # 获取 Agent 数据
            agent = AgentExecutor(f'http://{device.ip}:{device.agent_port}')
            try:
                raw_data = NvmeValidationService._fetch_data(agent, disk_name, test_type)
            finally:
                agent.close()

            # 加载规则并执行
            rules = NvmeValidationService.load_rules(test_type)
            results = NvmeValidationService.apply_rules(raw_data, rules)
            verdict = aggregate_verdict(results)

            # 写入结果
            test.result = [asdict(r) for r in results]
            test.verdict = verdict
            test.status = 'done'
            db.session.commit()

        except Exception as e:
            test = NvmeTest.query.get(test_id)
            test.status = 'failed'
            test.error = str(e)
            db.session.commit()
        finally:
            db.session.remove()


def _fetch_data(agent, disk_name, test_type):
    controller = NvmeService._extract_nvme_controller(disk_name)
    if test_type == 'identify':
        return agent.get_nvme_id_ctrl(f'/dev/{controller}')
    elif test_type == 'namespace':
        return agent.get_nvme_id_ns(f'/dev/{disk_name}')
    elif test_type == 'smart':
        return agent.get_smart(f'/dev/{disk_name}')
    elif test_type == 'error_log':
        return agent.get_nvme_error_log(f'/dev/{disk_name}')
    elif test_type == 'feature':
        # 依次读取 3 个 Feature
        data = {}
        for fid, label in [('0x06', 'fid_0x06'), ('0x02', 'fid_0x02'), ('0x0c', 'fid_0x0c')]:
            try:
                data[label] = agent.get_nvme_feature(f'/dev/{controller}', fid=fid)
            except Exception:
                data[label] = None
        return data
    elif test_type == 'fw_slot':
        return agent.get_nvme_fw_log(f'/dev/{controller}')
```

## §2 Identify 验证（VE-ID）

### §2.1 需求拆分

| 编号 | 最小需求 | 原子性验证 | 前置需求 |
|------|---------|-----------|---------|
| VE-ID-1 | 从 Agent 获取 id-ctrl 数据 | 独立上线：已有 API 可用 | 无 |
| VE-ID-2 | 加载 Identify 校验规则集 | 验收：YAML 文件可被 Service 正确解析 | VE-1 |
| VE-ID-3 | 执行校验 + 结果入库 + verdict | 验收：调用 API → 返回 PASS/FAIL + 各字段明细 | VE-ID-1, VE-ID-2 |
| VE-ID-4 | 前端展示校验结果表格（Pass/Fail 着色） | 验收：表格行按 pass/fail 红绿着色 | VE-ID-3 |

### §2.2 后端设计

**API 路径：**

```
POST /api/devices/<device_id>/nvme/validate    # 统一校验入口
GET  /api/nvme-tests/<test_id>                 # 查询单条结果
```

## §4 SMART 验证（VE-SM）

### §4.1 需求拆分

| 编号 | 最小需求 | 原子性验证 | 前置需求 |
|------|---------|-----------|---------|
| VE-SM-1 | SMART 数据已可通过 Agent 获取 | ✅ 已实现 | 无 |
| VE-SM-2 | 加载 SMART 校验规则（含阈值判定） | 验收：温度/磨损/媒体错误/关键告警/num_err_log_entries 均有规则 | VE-1 |
| VE-SM-3 | 执行校验 + 结果入库 + verdict | 验收：输出结构化 PASS/PARTIAL/FAIL | VE-SM-1, VE-SM-2 |
| VE-SM-4 | 前端展示：增加 num_err_log_entries / unsafe_shutdowns + Pass/Fail 标志 | 验收：SmartMonitorTab 中新增告警规则可见 | VE-SM-3 |

### §4.2 与现有告警规则的关系

V2 已实现的 `NvmeService.evaluate_alerts()` 6条告警规则与本版校验规则**高度重叠**，但定位不同：

| 维度 | 现有 alerts | 新校验规则 |
|------|-----------|-----------|
| 目的 | 实时监控告警 | 协议合规校验 |
| 输出 | 告警列表 | 结构化 Pass/Fail 报告 |
| 存储 | 不持久化 | 持久化到 nvme_tests |
| 新增字段 | — | num_err_log_entries, unsafe_shutdowns |

**决策（P1）：** 校验规则引擎**不依赖** evaluate_alerts，独立实现。原因：(1) 输出格式不同，合并引入适配复杂度；(2) 校验引擎需支持 YAML 配置化，现有告警是硬编码。

**决策（P4）：** SMART 校验完成后，SmartMonitorTab 中新增校验结果入口，**不修改**现有 CriticalAlertList 组件。

### §4.3 NvmeSmartData 模型变更

新增两个字段到 `nvme_smart_data` 表：

```sql
ALTER TABLE nvme_smart_data ADD COLUMN num_err_log_entries BIGINT DEFAULT 0;
ALTER TABLE nvme_smart_data ADD COLUMN unsafe_shutdowns BIGINT DEFAULT 0;
```

**兼容性：** 新增列 DEFAULT 0，无破坏性。已有数据自动填充默认值。

需同步修改：
- `NvmeSmartData.to_dict()` 增加 `num_err_log_entries`, `unsafe_shutdowns`
- `IngestService.ingest_nvme_smart()` 增加字段入库
- Agent `smart_collector.collect()` 确认原始数据包含这两个字段

## §5 Error Log 验证（VE-EL）

### §5.1 需求拆分

| 编号 | 最小需求 | 原子性验证 | 前置需求 |
|------|---------|-----------|---------|
| VE-EL-1 | Agent 端实现三步流程（读基准→触发错误→验证增加） | 验收：Agent 可返回 before/after/increased | Agent NVMe Collector 修改 |
| VE-EL-2 | Server 端接收 before/after 并存储 | 验收：nvme_test.result 包含三步数据 | VE-1 |
| VE-EL-3 | 前端对比展示 | 验收：展示触发前后条数+是否符合预期 | VE-EL-2 |

### §5.2 Agent 端三步流程

```
1. 读取基准 error log 条数
   命令：nvme error-log /dev/{device} -o json
   提取：len(error_log_entries)

2. 主动触发一次无效命令（产生错误）
   命令：nvme read /dev/{controller} -n 99 -c 1 -z 512
   预期：返回非零退出码，error log 条数 +1
   ⚠️ 无需 sudo，普通用户权限即可执行

3. 再次读取 error log 条数
   命令：同步骤1
   对比：after > before → PASS
```

**Agent 新增端点：**

```
POST /nvme/<device>/error-log-verify
```

**Agent 端返回：**

```json
{
  "before_count": 5,
  "after_count": 6,
  "increased": true,
  "trigger_error_output": "NVMe Status:INVALID_NAMESPACE..."
}
```

**兜底：** 若步骤2（触发错误命令）超时或执行失败（超时 15s），跳过触发，直接返回 `{"before_count": N, "after_count": N, "increased": false, "skipped": true, "skip_reason": "触发命令超时"}`，校验结果标记为 PASS 并 reason 注明"未触发验证"。

### §5.3 Server 端 AgentExecutor 扩展

```python
def verify_nvme_error_log(self, device: str) -> dict[str, Any]:
    response = self.session.post(
        f'{self.agent_url}/nvme/{device}/error-log-verify',
        timeout=30,
    )
    response.raise_for_status()
    return response.json()
```

### §5.4 Server 端校验规则

Error Log 的校验不在通用规则引擎的 YAML 中（因为 before/after 不是标准 JSON 字段比对），而是使用专用规则：

```python
# error_log 专用校验
def verify_error_log_result(verify_data: dict) -> list[CheckResult]:
    results = []
    before = verify_data.get('before_count', 0)
    after = verify_data.get('after_count', 0)
    increased = verify_data.get('increased', False)
    skipped = verify_data.get('skipped', False)

    if skipped:
        results.append(CheckResult(
            field='error_log_verify', value=f'skipped',
            check='错误日志增加验证', pass_=True,
            reason=f'跳过触发验证（{verify_data.get("skip_reason", "")}），结果为通过',
            level='warn',
        ))
    elif increased:
        results.append(CheckResult(
            field='error_log_verify', value=f'{before}→{after}',
            check='错误日志增加验证', pass_=True,
            reason=f'错误日志正确增加（{before}→{after}）',
            level='fail',
        ))
    else:
        results.append(CheckResult(
            field='error_log_verify', value=f'{before}→{after}',
            check='错误日志增加验证', pass_=False,
            reason=f'错误日志未增加，可能 error log 功能异常',
            level='fail',
        ))
    return results
```

## §10 方案对比与最佳实践选择

### §10.1 校验规则引擎：YAML 配置 vs 硬编码

| 维度 | YAML 配置 | 硬编码 if-else |
|------|-----------|---------------|
| 开发速度 | 慢（需定义解析器） | 快（直接写） |
| 可维护性 | 高（规则与代码分离） | 低（规则散落在代码中） |
| 灵活性 | 高（用户可自定义规则） | 低 |
| 调试复杂度 | 中（需追踪 YAML 解析） | 低 |

**决策（P0）：** 采用 YAML 配置。原因：(1) 产品规划中明确标注"可配置，后续功能复用"；(2) 6 个 test_type 共享引擎，硬编码会导致大量重复逻辑；(3) YAML 解析器只需一次开发，后续新增规则零代码改动。

### §10.2 Error Log 验证：三步流程 vs 纯展示

| 维度 | 三步流程（触发验证） | 纯展示（只读条数） |
|------|-------------------|-------------------|
| 验证价值 | 高（验证 error log 机制可用） | 低（只看到数据，未验证功能） |
| 实现复杂度 | 中（Agent 新增端点） | 低 |
| 安全风险 | 低（nvme read -n 99 无破坏性） | 无 |

**决策（P1）：** 采用三步流程。产品规划中明确描述了"读基准→触发→验证"三步流程，这是 NVMe 协议验证的核心价值点。

### §10.3 SMART 模型新增字段的入库时机

| 方案 | 说明 |
|------|------|
| A：立即修改 IngestService | 同时修改 Agent 端 smart_collector 确保上报 |
| B：仅修改校验引擎从 raw data 提取 | 不改入库链路 |

**决策（P1）：** 方案 A。`num_err_log_entries` 和 `unsafe_shutdowns` 是 NVMe SMART 日志的标准字段，入库后对 SmartMonitorTab 也有展示价值。入库后前端趋势图也能增加这两条线。

## §11 兜底策略

### VE-1 规则引擎

| 场景 | 兜底方式 | 最终状态 | 恢复周期 |
|------|----------|---------|---------|
| YAML 解析失败（格式错误） | try-except 捕获 → 降级为内置硬编码默认规则 → 日志 WARN | 校验可运行，但规则不可配置 | 1d |
| YAML 中 params 动态标记替换失败 | 回退到不替换，直接使用 params 原值 → 对应规则判定 FAIL | 部分规则可能误判 | 0.5d |

### VE-EL Error Log 三步验证

| 场景 | 兜底方式 | 最终状态 | 恢复周期 |
|------|----------|---------|---------|
| Agent 触发命令超时（15s） | 跳过触发步骤，返回 skipped=true | 结果标注"未验证"，verdict 不因此项 FAIL | 0d（自动恢复） |
| Agent 触发命令返回 0 退出码（未产生错误） | increased=false + reason 说明 | 判定 FAIL，但降级为 warn | 0.5d |

### VE-SM SMART 字段缺失

| 场景 | 兜底方式 | 最终状态 | 恢复周期 |
|------|----------|---------|---------|
| Agent 未返回 num_err_log_entries / unsafe_shutdowns | 这两个字段视为 null，对应校验规则 pass=true + reason="字段缺失，跳过校验" | 不影响整体 verdict | 0d（自动） |

### 核心安全网

**最坏情况下对已上线功能的影响为零。** 新增 `nvme_tests` 表、新增 API 路由、前端新增组件，均与现有 V1/V2 功能完全隔离。`NvmeSmartData` 模型新增列为 DEFAULT 0，无破坏性。
